# Transparent AI Apprentice Codex Plugin

This is the Codex plugin form of the project. It is not a standalone web app delivery. The plugin helps a human train an AI assistant through a direct teaching loop:

Human demonstrates -> Codex executes -> human corrects -> plugin extracts rule drafts -> human reviews -> future runs use reviewed memory.

The plugin keeps private chain-of-thought hidden. It exposes only public structured traces: steps, observed inputs, proposed action, rule candidates, confidence, validation, and human review points.

## Natural Human Communication

The packaged `warm-human-communication` skill applies to clarification, correction, failure, progress, technical, emotional, and casual replies. It answers the user's latest point first, keeps technical truth and review boundaries visible, and avoids customer-service scripts, process narration, generic comfort, sycophancy, false human biography, and emotional dependency.

```bash
node plugins/transparent-ai-apprentice/scripts/compile-human-communication-guidance.mjs --message "你怎么又理解错了" --context auto
node plugins/transparent-ai-apprentice/scripts/check-human-communication-style.mjs --text "draft reply" --context correction
npm run smoke:human-communication
```

The compiler emits response guidance rather than a canned final answer. The style checker is context-aware, so concise technical status language remains valid while the same changelog language can be flagged in a correction or emotional moment. Both are exposed in advanced MCP mode as `compile_human_communication_guidance` and `check_human_communication_style`.

## Separate Image, Office Text, And Engineering Software Masks

The original `create_transparent_sketch_overlay_kit` and `assets/mask-workbench` remain the restored engineering-image overlay. They do not contain a text/image/engineering switch. Office text uses the standalone `create_office_text_mask_workbench` plus `assets/text-mask-workbench`; engineering software uses `create_engineering_software_mask_workbench` plus `assets/engineering-software-mask-workbench`. The dedicated pages have different result formats and do not show each other's fields.

The two new workbenches still separate `change`, `protect`, and `reference` marks. Their `mingtu_surgical_edit_contract_v1` changes only selected targets, preserves every unmarked region, requires before/after comparison outside the mask, and blocks whole-artifact regeneration for a local issue unless the teacher chooses a separate regeneration candidate.

Text masks are not treated as image OCR. A Word edit binds to `paragraph:N`; an Excel edit binds to `SheetName!A1`. `surgical-office-text-edit.py` writes a separate DOCX/XLSX, requires one exact source-text match, never overwrites the source, and hashes every package part to prove only `word/document.xml` or the selected worksheet XML changed. The report narrows any visual spot check to the changed paragraph or cell instead of asking the user to reread the whole file.

```bash
npm run demo:engineering-mask
npm run demo:office-text-mask
npm run smoke:mask-workbench
npm run smoke:independent-masks
npm run smoke:office-surgical-edit
npm run smoke:rule-conflicts
```

Apparent learned-rule conflicts are handled through `resolve_learned_rule_conflicts`: current context, scope specificity, teacher-confirmed exceptions, evidence, confidence, priority, and risk are compared. The result records selected and suppressed rules plus a visible `apparent_rule_conflict` marker; underlying memory is never silently mutated or dropped.

## Image2 First-Generation Prompt Optimizer

The packaged `image2-prompt-optimizer` skill runs before a first Image2 generation. It routes to existing capabilities, separates confirmed facts from assumptions, compiles `mingtu_image2_initial_prompt_guidance_v1`, and blocks packaging generation when product type or authoritative dimensions are missing.

```bash
node plugins/transparent-ai-apprentice/scripts/compile-image2-initial-prompt.mjs --input request.json --output prompt-guidance.json
npm run smoke:image2-prompt-optimizer
```

The optional `IMAGE2_PROMPT_LIBRARY` can search the local 125,399-entry catalog for patterns. The large external database is not bundled and is never required: the plugin includes a deterministic fallback compiler and its own first-generation contract. Prompt retrieval is evidence only and cannot approve dimensions, skip teacher review, enable rules, or unlock packaging.

## Overall Direction: TLCL

The project is now intentionally converged on **Teachable Logic Contract Layer for AI Workflows** (`TLCL`, 鍙暀瀛﹂€昏緫濂戠害灞?. See `TLCL_OVERALL_DIRECTION.md`.

Transparent AI Apprentice is not a generic chatbot, a universal autonomous Agent, or a claim that AI can control all software. It is the trusted delivery layer for AI work: humans teach rules, strong models compile those lessons into contracts, lower-cost runtimes execute bounded work, deterministic validators judge results, and humans retain approval and responsibility.

First-stage development should prioritize the TLCL chain:

```text
Artifact Envelope -> Rule Card -> Rule DSL / Rule Package -> Validator Registry -> Validation Report -> Delivery / Execution Gate -> Audit Trail
```

The complete merged target and task list is in `FULL_TARGET_DIRECTION_AND_TASK_LIST.md`. The original user-requested lanes remain required: all-software low-token learning, transparent overlay, voice/text engineering control, numbered confirmation, detail-logic learning, RAG, rollback points, and existing-tool-first feasibility. The new direction does not delete them; it requires each lane to produce evidence that can enter the TLCL chain.

The model-cost strategy is explicit: highest-reasoning models learn, repair, and compile the full normative logic; medium-reasoning runtimes execute already-confirmed workflows under that contract; low-reasoning/tool layers handle fixed transforms and metadata work; teacher corrections, validator unknown/error, blocking failure, missing evidence, or rule conflicts escalate back to the senior compile layer. `create-tlcl-status-refresh` writes a review-only status packet and reasoning-tier contract for this boundary.

`create-tlcl-direction-operational-console` turns the current direction into one low-token teacher-facing operation package. It scans the existing TLCL/RAG return-chain gates, checks the reasoning-budget and provider-boundary surfaces, chooses the next review route, and writes `transparent_ai_tlcl_direction_operational_console_v1` plus Markdown and HTML start files. The route can only point to teacher route selection, RAG evidence attachment, high-reasoning contract repair, or reasoning-budget review before medium-runtime reuse; it does not invoke models, fetch RAG, execute target software, write memory, enable rules, unlock packaging, or claim completion. Advanced MCP exposes this as `create_tlcl_direction_operational_console`, and `smoke:plugin-tlcl-direction-operational-console` verifies default, RAG, correction, and runtime routes plus the no-op locks.

`create-tlcl-next-route-input-contract` consumes `transparent_ai_tlcl_direction_operational_console_v1` and writes `transparent_ai_tlcl_next_route_input_contract_v1`. It converts the selected route into a strict input checklist before any next route can be used: a launcher route needs a teacher route choice, a RAG route needs a TLCL packet plus reviewed RAG validation plus evidence-only teacher confirmation, a high-reasoning repair route needs a reviewed TLCL RAG evidence attachment plus retained rollback, and a medium-runtime route needs reasoning-budget review, reusable-workflow validation, and retained rollback. This closes the practical gap between "the console knows the next route" and "the next route is allowed to be prepared"; it writes JSON/Markdown/HTML only and keeps model calls, RAG fetch, next-tool execution, software execution, memory writes, rule enablement, packaging unlock, and completion claims locked. Advanced MCP exposes this as `create_tlcl_next_route_input_contract`, and `smoke:plugin-tlcl-next-route-input-contract` verifies missing-evidence and ready-input paths.

## Knowledge-Augmented RAG Research Direction

The project now tracks knowledge enhancement and RAG as a research lane for ordinary users to cultivate domain-specialized agents at low cost. See `KNOWLEDGE_AUGMENTED_RAG_RESEARCH_DIRECTION.md`.

## Feasibility-First Direction

Before choosing observation or execution tools, the plugin should profile how the teacher naturally teaches. Use `create_teacher_learning_method_profile` or let ordinary `teach_apprentice` route teacher-method messages into a `teacherMethodProfile` card. The profile writes `transparent_ai_teacher_learning_method_profile_v1`, preferred teaching modes, evidence order, question policy, correction policy, and suggested low-token next tools while keeping `ruleEnabled=false`, `accepted=false`, `technologyAccepted=false`, `packagingGated=true`, `fullContinuousRecording=false`, and `nativeUniversalExecution=false`.

Early development should prove the learning loop with existing technology. For visual teaching, the plugin should prefer existing drawing software or exported artifacts before building a custom canvas. For voice teaching, the plugin should prefer browser/system speech features before building a custom speech engine: Web Speech API dictation, `speechSynthesis` read-back, and manual transcript fallback are enough for feasibility. For work-along teaching in CAD/SolidWorks or other desktop software, the plugin should avoid continuous recording by default: watch log timestamps and log-tail deltas first, trigger screenshots only on errors, non-normal rebuild/export changes, teacher markers, or before/after state changes, then ask short teacher questions through voice or a floating browser overlay only when the event is ambiguous or could become reusable memory. For unfamiliar arbitrary software, first use the software capability profile to discover process/window metadata, log roots, Windows events, file deltas, and teacher markers, then use the universal observer kit to collect compact observations before screenshots; do not pretend every app exposes useful logs. For transparent sketch teaching, use the overlay kit to export 2D, perspective, and 3D spatial packets. When a reviewed overlay should become action, choose an existing software execution adapter first: browser automation for web apps, CLI/script commands, documented app APIs, file import/export, and only then supervised Windows UI automation. Use the supervised software action bridge to create a dry-run-first action plan and require explicit teacher confirmation before GUI events. For closed-loop learning, use `create_learning_workflow` to force the apprentice through causal modeling, boundary counterexamples, transfer reproduction, mastery gates, teacher approval, profile memory, and failure recovery before claiming the skill is reusable. Before risky changes, use rollback points and keep them until the teacher confirms the direction is correct. For conversational teaching, the plugin should capture the teacher's explanation, the apprentice's attempt, and the correction in the same session packet. The important proof is whether Codex can convert a human demonstration or teaching exchange into public trace rows, disabled rule drafts, counterexamples, and a replay plan.

The default user-facing path is `teach_apprentice`. A user should be able to give a goal, ordered action steps, pasted browser/recorder events, a whole `eventLog`/`screenEventLog`/`recordingEvents`/`traceEvents` export, a recording URL, before/after examples, whole `examples`/`trainingExamples`/`beforeAfterExamples` pairs such as `{input, output}`, a file, a teacher message, a teacher-method/adaptation request, a software auto-discovery/log-discovery request, a work-along request, a voice-teaching request, a learned-work request, a stop command, a correction, explicit approval, a natural "approve and remember/save" response, or a plain reply after reviewing a replay and let the plugin choose the next safe step. When the teacher says they prefer examples, overlay sketches, voice, logs, fewer questions, or correction-first review, `teach_apprentice` can create a `teacherMethodProfile` card with `transparent_ai_teacher_learning_method_profile_v1` and suggested next tools before spending observation or screenshot budget. When a visual teacher has no file yet, `teach_apprentice` can create a small existing-tool starter package with editable draw.io, Excalidraw, and Mermaid files plus the next teaching calls. When a teacher asks to automatically discover logs for an unfamiliar app, `teach_apprentice` can create `softwareCapabilityProfile` paths for a read-only probe, observation plan, and next universal observer arguments before screenshots. When a teacher wants the apprentice to learn beside CAD/SolidWorks or another app, `teach_apprentice` can create a low-token work-along kit with a PowerShell watcher, triggered screenshot policy, browser overlay, source map, and `transparent_ai_workalong_observation_v1` template. When a teacher says the next task is something the apprentice learned before, `teach_apprentice` can run approved profile memory and return a `learnedWork` card instead of asking for a new demonstration. When the teacher says "婵犻潧鍊归崹婵堣姳閿熺姴绀嗛柟宄扮灱缁?, "闂佺绻愰悧鍡樼?, or "stop here", `teach_apprentice` stops without generating more or writing memory. When a voice teacher has not started capture yet, `teach_apprentice` can create a browser voice kit that asks the first-use tone preference, captures dictation or manual transcript text, speaks timely confirmations with `speechSynthesis`, and outputs a `transparent_ai_voice_teaching_turn_v1` packet to paste back. When the teacher has a Jam/browser recorder/Playwright trace link but no exported event JSON, pass `recordingUrl` to `teach_apprentice`; when the teacher has exported events, paste the whole export object as `eventLog` or `traceEvents`; when the teacher has text examples, paste them as `examples`.
